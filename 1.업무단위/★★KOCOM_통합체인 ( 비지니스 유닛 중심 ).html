단 한 장의 완성 체인.

KOCOM 시스템은 목록(📋) → 입력(✏️) → 처리(⚡) → DB 프로시저(🧩) → 테이블(🗄️) 5단계 사다리입니다. 206개 화면이 45개 업무 단위로, 17개 핵심 DB 프로시저가 사용자 액션·트리거·스케줄 Job·내부 호출 4경로로 연결됩니다. 어느 방향으로 세어도 테이블 쓰기 총합이 같으며, 프로시저 레이어도 상류(호출 경로) ↔ 하류(쓰기 테이블)가 추적됩니다.

📖 용어 정의 — 업무 단위 (문서 내에서 "BU" = Business Unit 으로 약칭): 서로 이어지는 화면들(목록+입력+처리+테이블)을 하나의 업무 묶음으로 정의한 단위. 예) 사원 관리 업무단위 = admin_user_list.php + admin_user_mode.php + admin_user_proc.php + CS사원·CS로그인 테이블.

업무 1건이 화면부터 테이블까지 가는 5단계

예시: 자재 신청 승인 — 센터가 신청한 자재를 본사가 승인해서 창고 재고를 차감하는 업무. 각 단계의 색이 카드 전반에서 그대로 쓰입니다.

1
📋
목록
승인 대기 중인 자재 신청들을 조회·검색하는 화면
예시 파일 store_request_list.php
→
2
✏️
입력
승인 수량·상태·비고를 입력하는 폼 화면
예시 파일 store_request_release.php
→
3
⚡
처리
입력된 POST를 받아 SQL을 실행하고 DB 프로시저를 호출하는 PHP 핸들러
예시 파일 store_request_release_proc.php
→
4
🧩
DB 프로시저
깊은 DB 로직. 13개 인자를 받아 양품/재생/불량 3단 재고를 한 번에 차감
예시 오브젝트 P_CS창고재고변경
→
5
🗄
테이블
최종 영구 저장. 자재신청품목·창고재고 두 테이블이 갱신됨
대상 테이블 CS자재신청품목, CS창고재고
🎨
색이 곧 단계입니다

파랑→청록→주황→보라→빨강. 이 순서만 기억하면 업무단위 카드·감사 뷰·프로시저 뷰 어디서든 단계를 식별할 수 있어요.

⚖️
숫자가 정확히 같아야 정합

화면 쪽에서 센 DB 쓰기 횟수 = 테이블 쪽에서 센 횟수. 이 등식이 무너지면 데이터 오류입니다. 정합성 감사 탭에서 실시간 확인.

🔗
4단계는 때때로 생략

사원 등록처럼 _proc.php가 직접 SQL을 실행하는 경우엔 DB 프로시저 없이 3→5로 바로 갑니다. 카드에 "직접 SQL만 수행"으로 표시.
